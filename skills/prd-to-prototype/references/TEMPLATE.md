# 스펙독 정규화 템플릿

> **목적**: PRD 원문을 프로토타입 구현 가능한 형태로 정규화한다.
> **원칙**: PRD에 명시된 내용만 옮긴다. 해석·추론·판단을 더하지 않는다.
> PRD에 없는데 구현에 필요한 정보는 "불명확한 항목"에 넣어 사용자에게 확인받는다.

---

## 페이지 템플릿

```markdown
## [페이지명]

- 페이지 유형: List / Form / Detail
- 경로: /
- 브레드크럼: >
- 사이드바 메뉴: {1depth} > {2depth 라벨}
- 헤더 브레드크럼: {parent} > {current}

### 검색 필터
| 라벨 | 컴포넌트 | 옵션 | 필수 |
|------|----------|------|------|
| | {허용값만} | {PRD 원문만} | Y/N |

### 테이블 컬럼
| 컬럼명 | 셀 유형 | 비고 |
|--------|---------|------|
| | {허용값만, 기본=Text} | {PRD 원문만} |

### 정렬
- PRD 정렬 언급: 있음 / 없음
- 있음 → 정렬 대상 컬럼과 옵션을 PRD 원문대로 기재
- 없음 → "없음"으로 기재하고 구현하지 않음

| 기본 정렬 | 옵션 |
|-----------|------|
| {컬럼명} {오름차순/내림차순} | {PRD에 나열된 정렬 옵션} |

### 폼 필드
| Lv1(섹션) | Lv2(그룹) | Lv3(필드) | 컴포넌트 | 필수 |
|-----------|----------|-----------|----------|------|
| | | | {허용값만} | Y/N |

### 모달
| 모달명 | 유형 | 트리거 | 필드 목록 |
|--------|------|--------|-----------|
| | Info(600px) / Alert(400px) | {PRD 원문의 트리거 조건} | {필드 나열} |

- 유형 판단 기준: 폼 입력이 있으면 Info, 확인/취소만이면 Alert
- PRD에 모달 언급이 없으면 "없음"으로 기재

### 프레임
| 프레임명 | 위치 | 내용 |
|----------|------|------|
| | | {PRD 원문만} |

- PRD에 프레임/상태 언급이 없으면 "없음"으로 기재

### 불명확한 항목
| # | 항목 | 불명확 사유 | 제안 (선택) |
|---|------|------------|-----------|
| | | {왜 판단할 수 없는지} | {있다면 선택지 제시} |
```

## 작성 규칙

### 비고 컬럼 규칙
- PRD 원문에 적힌 내용만 옮긴다
- 원문에 없는 해석(계산 방식 추론, 스타일 판단 등)을 넣지 않는다
- 원문의 설명이 길면 핵심만 축약하되, 의미를 변경하지 않는다

### 필수 여부 판단
- PRD에 "필수", "required", "*" 등이 명시된 경우에만 `Y`
- 명시가 없으면 `N`
- 판단이 애매하면 → 불명확한 항목

### PRD 내 동일 필드 참조 규칙
- 같은 필드가 PRD의 여러 페이지(등록/상세/수정 등)에 걸쳐 나올 때, 한쪽에서 컴포넌트 유형이 명시되고 다른 쪽에서는 생략되어 있으면, 명시된 쪽의 스펙을 따른다
- 양쪽 모두 생략이면 불명확한 항목으로 분류한다

### 옵션 값 기재
- PRD에 나열된 옵션만 기재한다
- 기본값이 명시된 경우 `(기본값)`으로 표기
- PRD에 옵션 목록이 없으면 비워두고 불명확한 항목에 추가

---

## 교차 검증 (정규화 완료 후, 사용자 확인 전에 수행)

1. **필드 누락 검증**: 정규화된 스펙독의 필드 목록을 PRD 원문과 대조하여 빠진 필드가 없는지 확인
2. **관련 페이지 섹션 구조 정렬**: 같은 도메인의 등록/상세/수정 페이지가 있으면, Lv1/Lv2 섹션 구조와 필드 순서를 통일한다

## 허용값 정의 (부록)

> 아래 허용값은 가이드 `.claude/designguide/20-components.md` 섹션 8에 정의된 컴포넌트/셀 유형에서 도출한다.
> **허용값에 없는 값을 사용하려면 → 불명확한 항목으로 이동하여 사용자에게 확인받는다.**

### 검색 필터 — 컴포넌트 허용값

| 컴포넌트 | 가이드 근거 | 설명 |
|----------|-----------|------|
| `TextInput` | 8-2 | 자유 텍스트 입력 (w-240px) |
| `Select` | 8-4 | 단일 선택 드롭다운 (옵션 목록 고정) |
| `Dropdown` | 8-5 | 검색 가능한 드롭다운 (파트너/셀러샵 등 대량 목록) |
| `Radio` | 8-6 | 택1 선택 (가로 배치, gap 40px) |
| `Checkbox` | 8-6 | 다중 선택 (가로 배치, gap 24px) — **"전체" 처리는 아래 규칙 참조** |
| `TextArea+Radio` | 10-1, 8-6 | 복수검색 전용 (Radio로 검색 기준 택1 + Textarea로 값 입력, 세로 배치 gap 8px) |
| `DatePicker` | Storybook | 날짜 선택 |

#### Checkbox "전체" 옵션 처리 규칙 (필수)

> PRD에 체크박스 옵션을 나열할 때 "전체"가 어떻게 표기됐는지에 따라 구현이 달라진다. **PRD 원문 그대로 반영하는 것이 원칙.**

| PRD 표기 패턴 | 스펙독 정규화 | 구현 동작 |
|--------------|--------------|----------|
| `"전체, A, B"` 또는 `"전체/A/B"` 처럼 **"전체"가 명시적으로 나열**됨 | "전체"를 **명시적 체크박스 옵션**으로 포함 | - "전체" 체크 시: 다른 모든 옵션 선택 + "전체" 체크<br>- 다른 옵션 전부 해제 시: "전체" 자동 체크<br>- 다른 옵션 하나라도 해제 시: "전체" 자동 해제<br>- 초기값이 "전체"면 모든 체크박스(전체 포함) 선택 상태로 시작 |
| `"A, B"` (전체 미언급) | 옵션만 나열, "전체" 추가 금지 | 빈 배열 = "전체" 의미로 해석. UI에 "전체" 체크박스 표시 안 함 |
| `"기본값: 전체"` 언급만 있고 옵션 목록에 "전체" 없음 | 옵션만 나열 | 초기 상태 = 모든 옵션 체크 (빈 배열이 아니라 전체 선택 상태) |

**❌ 금지 패턴:**
- ❌ PRD에 "전체"가 나열돼 있는데 "암묵적으로 빈 배열=전체 의미"로 재해석해서 옵션 축소
- ❌ PRD에 "전체" 없는데 UX 개선 차원에서 임의로 "전체" 추가

## 불명확한 항목으로 보내야 하는 경우

| 상황 | 예시 |
|------|------|
| 컴포넌트/셀 유형이 허용값에 없음 | PRD에 "토글", "슬라이더", "칩" 등 |
| PRD에 옵션 목록 없이 "선택" 만 언급 | "배송타입 선택" (옵션 미나열) |
| 필수 여부 판단 불가 | 별표도 없고 "필수" 언급도 없는 필드 |
| 셀 렌더링 방식 불명 | "상태" 컬럼인데 활성/비활성이 아닌 다른 값 |
| 계산 로직 미정의 | 컬럼은 있는데 계산 방식 설명 없음 |
| 데이터 소스 미정의 | 값이 어디서 오는지 설명 없음 |
| 페이지네이션/정렬 미정의 | 목록인데 페이지네이션 언급 없음 |
