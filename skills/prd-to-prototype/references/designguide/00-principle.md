### 0-2. 컴포넌트 스펙 교차 검증

> ⚠️ 페이지(섹션 10)를 구현할 때 **모든 UI 요소는 섹션 8(컴포넌트) 스펙을 그대로 사용해야 한다.**
> 시각적 판단("이건 좀 커 보이는데", "이건 짧은 값이니까 줄이자")으로 너비·높이·여백을 **임의 변경 금지.**
> 페이지 레이아웃 규칙과 컴포넌트 스펙이 충돌하면, **섹션 8 컴포넌트 스펙이 우선한다.**

#### 0-2-1. 구현 전 교차 검증 체크리스트

> 페이지 코드를 작성하기 **전에**, 사용할 컴포넌트마다 섹션 8의 해당 하위 절을 반드시 다시 읽는다.
> 0-2는 **"어느 절을 참조해야 하는지"** 만 제시한다. 실제 값/스펙은 섹션 8의 해당 절이 유일한 소스다.

| 요소 | 섹션 8 참조 절 |
|------|--------------|
| Button | 8-1 |
| Input | 8-2 |
| Textarea | 8-3 |
| Select | 8-4 |
| Dropdown | 8-5 |
| Radio / Checkbox | 8-6 |
| Table | 8-7 |
| Pagination | 8-8 |
| Modal | 8-9 |
| Snackbar | 8-10 |

#### 0-2-2. 교차 검증 절차

> 모든 input/select/button/table 등을 렌더링하는 지점마다 아래 절차를 거친다.

1. 이 요소가 속한 컴포넌트 카테고리를 결정 (예: "짧은 숫자 입력" → Input)
2. 섹션 8에서 해당 컴포넌트 스펙 테이블을 **다시 읽음** (기억에 의존 금지)
3. 스펙 표의 **"허용되는 값"** 만 사용
4. 페이지 레이아웃상 꼭 다른 값이 필요하다 판단되면 → **코드를 쓰기 전에 사용자에게 확인 요청**, 자의적으로 변경하지 않음

#### 0-2-3. 근본 원칙

> **섹션 8에 정의된 값만 사용한다. 정의되지 않은 값이 필요하면 구현 전에 사용자에게 확인한다.**
> 이유가 무엇이든 — 시각적 판단, UX 개선, 구분 목적 등 — 섹션 8에 없는 색상·크기·스타일을 임의로 만들지 않는다.

#### 0-2-4. 허용되는 예외

> 섹션 8 스펙에서 벗어나려면 **반드시 아래 중 하나에 해당**해야 한다.

- Spec doc에 구체적 너비/높이가 명시된 경우
- 사용자가 대화 중에 명시적으로 다른 값을 지시한 경우
- 섹션 8에 "가변" 또는 "유동"으로 명시된 속성

> 위에 해당하지 않으면 섹션 8 스펙 **그대로** 사용. 임의 변경 발견 시 즉시 스펙 값으로 되돌린다.

#### 0-2-5. 자체 검증 (구현 후)

> 페이지 작성 후, 커밋 전에 아래 절차를 수행한다.

1. 페이지에서 사용된 컴포넌트 종류를 모두 나열한다 (Input, Select, Button, Table, Modal...)
2. 각 컴포넌트에 대해 섹션 8의 해당 절을 **다시 읽고** 실제 코드 값이 그대로인지 대조한다
3. 같은 컴포넌트가 페이지 내에서 일관된 스펙으로 구현되었는지 확인한다
4. 하나라도 섹션 8 스펙과 다르면 섹션 8 값으로 되돌린다 (0-2-4 예외에 해당하지 않는 한)

### 0-2-6. 관련 페이지 일관성 규칙

> ⚠️ 동일 도메인(예: SKU, 파트너, 옵션물류 등)의 **등록 / 상세 / 수정 / 조회** 등 관련 페이지를 새로 만들 때,
> 이미 구현된 페이지의 **섹션 분류 · 필드 그룹핑 · 네이밍**을 기준으로 삼는다.

#### 규칙

1. **기존 구현 우선 확인**: 신규 페이지 코드를 작성하기 **전에**, 같은 도메인의 기존 페이지(`src/app/` 하위)를 먼저 읽고 섹션 구조를 파악한다
2. **스펙독과 기존 구현이 충돌할 때**: 스펙독의 섹션 분류(그룹핑, 네이밍)와 기존 구현이 다르면 **기존 구현을 따른다**
3. **차이점 보고**: 스펙독과 기존 구현 사이에 분류 차이가 있을 경우, 구현 전 사용자에게 차이점을 보고하고 기존 구현 기준으로 진행함을 알린다
4. **필드 추가/제거는 스펙독 따름**: 섹션 분류(그룹핑)는 기존 구현을 따르되, 어떤 필드가 포함되는지(추가/제거/변경)는 스펙독이 우선한다

#### 예시

> SKU 등록이 `기본정보 / SKU정보 / 박스정보 / 추가정보`로 이미 구현되어 있는데,
> SKU 상세조회 스펙독이 `기본정보 / SKU 상태 / 박스정보 / 대표소재 / 추가정보`로 분류되어 있다면:
> → 기존 SKU 등록의 섹션 분류(`기본정보 / SKU정보 / 박스정보 / 추가정보`)를 따르고,
> → 필드 차이(읽기전용 여부, 새 필드 추가 등)는 스펙독대로 반영한다

#### ❌ 금지 패턴

- ❌ 기존 관련 페이지를 확인하지 않고 스펙독의 섹션 분류를 그대로 사용하여 같은 도메인 내 페이지 간 구조가 불일치하는 것
- ❌ 차이를 발견했는데 보고 없이 임의로 한쪽 기준을 선택하는 것

### 0-3. 신규 페이지 추가 시 공유 컴포넌트 동반 수정

> ⚠️ 신규 라우트(`src/app/.../page.tsx`)를 만들 때 페이지 파일만 추가하면 안 된다.
> 사이드바·헤더 같은 공유 컴포넌트에 해당 경로 엔트리를 반드시 함께 등록한다.
> 누락 시 페이지는 "URL 직접 입력으로만 접근 가능"하거나 "헤더가 다른 페이지 이름으로 잘못 표시"되는 미완성 상태가 된다.

#### 0-3-1. 동반 수정 체크리스트

> ⚠️ 신규 라우트 추가 시 아래 항목을 모두 확인하지 않으면 페이지 구현은 완료된 것으로 간주하지 않는다.

| # | 파일 | 확인 / 수정 내용 | 누락 시 증상 |
|---|------|----------------|-------------|
| 1 | `src/components/Header.tsx` | `BREADCRUMB_MAP`에 신규 경로 키 추가 (`{ parent, current }`) | 헤더 브레드크럼이 fallback 값(예: 다른 페이지 이름)으로 잘못 표시 |
| 2 | `src/components/Sidebar.tsx` | `MENU_ITEMS`의 해당 1depth 메뉴 `submenu`에 신규 항목 추가, `href` 연결, 현재 페이지일 때 `active: true` 처리 | 사이드바 메뉴에서 진입 불가, active 하이라이트 없음 |
| 3 | 기존 라우트와의 충돌 | 신규 경로 prefix가 기존 라우트와 겹치지 않는지 확인 | 의도하지 않은 페이지 매칭 또는 404 |
| 4 | 페이지 도달성 검증 | dev 서버에서 **URL 직접 입력이 아닌 사이드바 메뉴 클릭으로** 접근하여 헤더 브레드크럼이 올바르게 표시되는지 확인 | "URL 치면 열리지만 메뉴에서는 못 찾음" 상태로 출시 |

#### 0-3-2. 근본 원칙

- **하나의 페이지 = 라우트 + 사이드바 진입점 + 헤더 브레드크럼**, 세 가지가 모두 연결된 상태가 "완성"이다.
- 공유 컴포넌트(Header/Sidebar)에 하드코딩된 맵(`BREADCRUMB_MAP`, `MENU_ITEMS` 등)이 있으면 **신규 경로 엔트리를 추가하지 않는 한 동작이 보장되지 않는다.** 코드를 작성하기 전 반드시 두 파일을 먼저 열어 구조를 확인할 것.
- "요청은 페이지만 만들어달라였다"는 이유로 공유 컴포넌트 수정을 건너뛰지 않는다. 사용자에게 전달할 최종 결과물은 **네비게이션을 통해 도달 가능한, 헤더가 올바른 페이지**이다.

#### 0-3-3. Do / Don't

✅ 페이지 파일 작성 → `Header.tsx`의 `BREADCRUMB_MAP` 업데이트 → `Sidebar.tsx`의 `MENU_ITEMS` 업데이트 → dev 서버에서 사이드바 메뉴 클릭으로 진입 검증
❌ 페이지 파일만 작성 후 "URL로 접근하면 열린다"로 종료
❌ "공유 컴포넌트 수정은 작업 범위 외"라고 판단하여 동반 수정을 생략

#### 0-3-4. 코드 예시

```tsx
// src/components/Header.tsx — BREADCRUMB_MAP에 신규 경로 추가
const BREADCRUMB_MAP: Record<string, { parent: string; current: string }> = {
  "/option-logistics/register": { parent: "풀필먼트 관리", current: "옵션물류정보 등록" },
  "/partner-search": { parent: "풀필먼트 관리", current: "파트너 조회" },
  "/sku/register": { parent: "풀필먼트 관리", current: "SKU 등록" }, // ← 신규 페이지 추가
};
```

```tsx
// src/components/Sidebar.tsx — MENU_ITEMS의 해당 1depth 서브메뉴에 추가
{
  label: "풀필먼트 관리", href: "#", active: true,
  submenu: [
    { label: "파트너 조회", href: "/partner-search" },
    { label: "SKU 등록", href: "/sku/register", active: true }, // ← 신규 페이지 추가 + 현재 페이지일 때 active
    { label: "옵션물류정보 등록", href: "/option-logistics/register" },
  ],
},
```

> 프로토타입 변환 스킬(`/prd-to-prototype`)에서 Sidebar/Header 연동도 이 규칙을 그대로 따른다. 충돌 시 0-3이 우선한다.

### 0-4. 인터랙션 구현 (프레임과 분리)

> ⚠️ **프레임 스위처(0-7)는 "상태 프리뷰"이지 "인터랙션 구현"이 아니다.**
> 아래 인터랙션은 프레임과 무관하게 **항상 동작**해야 한다.
> 프레임 = "각 상태의 정적 스냅샷" / 인터랙션 = "실제 동작하는 클릭 흐름"
> 둘 다 구현되어야 프로토타입이 완성이다. 프레임만 있고 인터랙션이 비어있으면 **미완성**이다.

#### 0-4-1. 필수 인터랙션 목록

| 인터랙션 | 필수 여부 | 구현 방식 |
|---------|--------|--------|
| Select 드롭다운 열기/닫기/선택 | 필수 | `isOpen` 상태 + 트리거 `onClick` + 항목 `onClick` (→ 섹션 8-4) |
| Input 포커스/입력 | 필수 | `value` + `onChange` 바인딩 |
| Radio 선택 | 필수 | `value` + `onChange`, 클릭 시 즉시 state 업데이트 |
| Textarea 입력 + 글자수 카운터 | 필수 | `value` state 바인딩, 카운터는 state 기반 실시간 계산 |
| 등록/수정 버튼 | 필수 | **항상 Confirm 모달 오픈** (프레임 조건부 금지) |
| 모달 취소 버튼 | 필수 | 모달 닫기 |
| 모달 확인/등록/수정 버튼 | 필수 | 모달 닫기 → **Snackbar 3초 표시** → (선택) 폼 초기화 또는 다음 화면으로 이동 |
| 취소 버튼 (페이지 하단) | 필수 | 동작 가능한 stub (`alert` 금지) |

#### 0-4-2. 등록 흐름 예시

```tsx
const [showConfirm, setShowConfirm] = useState(false);
const [snackbar, setSnackbar] = useState(false);

const handleSubmit = () => setShowConfirm(true);
const handleConfirm = () => {
  setShowConfirm(false);
  setSnackbar(true);
  setTimeout(() => setSnackbar(false), 3000);
};

<button onClick={handleSubmit}>등록</button>

{showConfirm && <ConfirmModal onCancel={() => setShowConfirm(false)} onConfirm={handleConfirm} />}
{snackbar && <Snackbar message="등록되었습니다." />}
```

#### 0-4-3. ❌ 금지 패턴

- ❌ 등록 버튼 `onClick`을 프레임 조건부로 만드는 것 (`onClick={() => isModal && setShowModal(true)}`) — 프레임에서 Modal을 선택해야만 모달이 열리는 잘못된 구현
- ❌ Select 트리거 `<div>`에 `onClick` 없이 정적 렌더링 (클릭해도 안 열림)
- ❌ 드롭다운 항목이 아예 없거나, `onClick` 없이 렌더링
- ❌ 스낵바를 구현하지 않고 "Modal 프레임에서 시각만 보여주면 됨"으로 끝내는 것
- ❌ `alert()` / `console.log()` 로 인터랙션 대체

### 0-5. SVG 아이콘 사용 규칙

> ⚠️ 모든 SVG 아이콘은 반드시 `src/components/Icons.tsx`에 정의된 컴포넌트를 import하여 사용한다.

**필수:**
- 아이콘이 필요하면 `@/components/Icons`에서 named import
- 새 아이콘이 필요하면 Icons.tsx에 먼저 named export 컴포넌트로 정의한 후 import하여 사용
- 아이콘별 export명·사용처·사이즈는 `10-design-tokens.md` 섹션 5 매핑 테이블 참조

**❌ 금지 패턴:**
- ❌ 인라인 `<svg>` 태그를 컴포넌트 파일에 직접 삽입
- ❌ SVG 아이콘을 텍스트 문자(∧, ∨, >, ✓ 등)로 대체
- ❌ SVG 아이콘을 CSS(`border-radius`, `background` 등)로 대체 구현
- ❌ SVG 아이콘을 유니코드 이모지로 대체

```tsx
// ❌ 금지: 인라인 SVG
<svg width="24" height="24" viewBox="0 0 24 24">...</svg>

// ✅ 필수: Icons.tsx에서 import
import { CloseIcon } from "@/components/Icons";
<CloseIcon />
```

### 0-7. 프로토타입 프레임 스위처 스펙

> 프로토타입 구현 시, PRD에서 여러 프레임/상태를 요구하면 **상단 플로팅 스위처**를 반드시 포함한다.
> Sidebar, Header 등 기존 레이아웃의 position/z-index에 영향을 주지 않는 오버레이 형태로 구현한다.
> 프레임 스위처는 **시각적 상태 프리뷰 전용**이다. 실제 인터랙션(Select 열기, 등록 흐름, 스낵바 등)은 **0-4 참조**.

**스펙**

| 속성 | 값 |
|------|-----|
| 위치 | `fixed top-[24px] left-1/2 -translate-x-1/2` |
| 배경 | #2F3438 |
| Border Radius | 12px |
| Shadow | `0px 4px 24px rgba(0,0,0,0.25)` |
| Padding | 6px |
| 버튼 간격 | 4px |
| z-index | 100 |
| 활성 버튼 | bg #0AA5FF, text white, radius 8px |
| 비활성 버튼 | bg transparent, text #828C94, hover: text white + bg #3D4348 |

**코드 예시**

```tsx
{/* 상단 플로팅 프레임 스위처 — 레이아웃에 영향 없음 */}
<div className="fixed top-[24px] left-1/2 -translate-x-1/2 bg-[#2F3438] rounded-[12px] shadow-[0px_4px_24px_rgba(0,0,0,0.25)] flex items-center gap-[4px] p-[6px] z-[100]">
  {frames.map((item, i) => (
    <button
      key={i}
      onClick={() => setFrame(i + 1)}
      className={`px-[16px] py-[8px] text-[13px] border-none cursor-pointer rounded-[8px] transition-colors ${
        frame === i + 1
          ? "bg-[#0AA5FF] text-white"
          : "bg-transparent text-[#828C94] hover:text-white hover:bg-[#3D4348]"
      }`}
    >
      {item.label}
    </button>
  ))}
</div>
```

**❌ 금지 패턴**
- ❌ 레이아웃 내부에 삽입 (콘텐츠 영역을 밀어냄)
- ❌ 레이아웃 내부에 탭으로 삽입하여 Sidebar/Header의 position/offset을 변경하는 것
- ✅ 반드시 `fixed` + `top-[24px]` + `z-[100]` 오버레이로 구현

**프레임 ↔ 모달 상태 동기화 규칙**

> 모달이 포함된 프레임이 있을 때, 프레임 전환과 모달 open/close 상태를 반드시 동기화한다.

```tsx
// 프레임 2가 "모달 열린 상태"일 때
const [isOpen, setIsOpen] = useState(frame === 2);

// 프레임 전환 시 모달 상태 동기화
useEffect(() => {
  setIsOpen(frame === 2);
}, [frame]);

// 닫기 버튼
<button onClick={() => {
  setIsOpen(false);
  setFrame(1); // 이전 프레임으로 복귀
}}>
```

**⚠️ 모달 내 폼 값 바인딩 규칙**

> 모달 내 수정 가능한 필드(Textarea, Input 등)의 `value`는 반드시 **state 변수**에 바인딩한다.
> 프레임 분기로 원본 데이터를 직접 넣으면 입력이 불가능해진다.

```tsx
// ❌ 금지: 프레임 분기로 고정값 바인딩 → 입력 불가
value={isEditFrame ? target.memo : editMemo}

// ✅ 필수: 항상 state 바인딩 + 프레임 전환 시 useEffect에서 초기값 세팅
value={editMemo}

useEffect(() => {
  if (frame === 5) { setEditMemo(target.memo); }
}, [frame]);
```

### 0-8. 프레임 자동 생성 규칙

> PRD에 "프레임 구성"이 명시되지 않더라도,
> 아래 조건에 해당하면 프레임 스위처를 자동으로 생성한다.

**자동 생성 조건**

- 필수 필드가 1개 이상 있으면 → Empty + Filled + Validation Error 프레임 생성
- 등록/수정/삭제 CTA가 있으면 → Confirm Modal 프레임 추가
- 상태 변화가 있으면 (예: 승인 전/후) → 상태별 프레임 추가

**기본 프레임 세트 (Form Page)**

| 프레임 | 조건 |
|--------|------|
| Empty | 항상 |
| Filled | 항상 |
| Validation Error | 필수 필드 존재 시 |
| Confirm Modal | 등록/수정/삭제 CTA 존재 시 |

**기본 프레임 세트 (List Page)**

| 프레임 | 조건 |
|--------|------|
| Default | 항상 |
| Empty State | 항상 |
| Filtered | 검색 필터 존재 시 |
